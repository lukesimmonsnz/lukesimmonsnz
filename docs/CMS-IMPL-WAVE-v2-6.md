# CMS-IMPL-WAVE-v2-6 — Template slot promotion (header.md + footer.md)

**Status:** design artefact — wave v2-6 of `CMS-SPEC-v2.md` §6.
**PI directive (2026-05-02):** authored in parallel; mechanical.
**Stage gating (V2-5 ratification):** header + footer first; sidebar
deferred to a v2-6b sub-wave.

---

## 1. The promotion

Currently `templates/_partials/nav.html` and
`templates/_partials/footer.html` are Jinja-only — code-level edits
required. v2 promotes them to PI-editable Markdown slots:

```
content/_theme/
  header.md        ← logo + nav (rendered into <header>/<nav>)
  footer.md        ← copyright + social links + meta (rendered into <footer>)
```

The Jinja base template's `{% include "_partials/nav.html" %}` is
preserved for backward-compatibility; the partial itself becomes a thin
shell that calls a new Jinja macro `{% slot_render('header') %}`.

---

## 2. The mathematical decomposition

Define $S$ = set of editable slots, $\rho_s : \text{slot} \to \text{HTML}$
the slot-render function, $C_s$ the slot's MD source content,
$\mu$ the standard MD-to-HTML pipeline, $\tau_s$ the slot's
template-shell partial.

$$\rho_s(C_s) = \tau_s(\mu(C_s))$$

The slot's identity round-trip property: editing $C_s$ via the CMS and
re-rendering produces only the diff at $\rho_s$; sibling pages and
unrelated slots are unaffected. This is the v1 page-publish locality
property lifted to slots.

## 3. New page kind dispatch

Resolver gains two URL patterns. They map to the existing
`DIRECT_MD_PROJ` projection — a slot is just a DIRECT_MD page rooted at
`content/_theme/`:

```python
# delta to blueprints/admin/cms/resolver.py

THEME_SLOTS: frozenset[str] = frozenset({"header", "footer"})  # "sidebar" later

def _build_theme_slot(match: re.Match) -> PageRef | None:
    slot = match.group("slot")
    if slot not in THEME_SLOTS:
        return None
    src = _CONTENT / "_theme" / f"{slot}.md"
    if not src.exists():
        return None
    return PageRef(
        url=f"/_theme/{slot}/",
        kind=PageKind.DIRECT_MD,
        source_paths=(src,),
        projection=DIRECT_MD_PROJ,
        lock=LockKind.EDITABLE,
        title=f"Theme: {slot}",
    )


# Append to _PATTERNS BEFORE the catch-all settings pattern:
(re.compile(r"^/_theme/(?P<slot>[\w-]+)/$"), _build_theme_slot),
```

The URL `/_theme/header/` is editor-only — there is no public-facing
page at that URL. It is a synthetic addressable surface for the
editor.

## 5. Render integration

The base template (`templates/base.html`) currently inlines partials:

```jinja
{% include "_partials/nav.html" %}
...
{% include "_partials/footer.html" %}
```

`nav.html` and `footer.html` are rewritten to consume the slot's
rendered HTML, which lives at the edited path's "rendered cache":

```jinja
{# templates/_partials/nav.html — rewritten #}
<nav class="site-nav">
  {{ slot_render('header') | safe }}
</nav>
```

`slot_render(name)` is a new Jinja global registered in `app.py`:

```python
# delta to app.py

from blueprints.admin.cms.theme_slots import slot_render as _slot_render

def create_app() -> Flask:
    app = Flask(__name__)
    ...
    app.jinja_env.globals['slot_render'] = _slot_render
```

`slot_render(name)` reads the rendered cache, falling through to a
runtime render when the cache is missing or stale. Rendered cache lives
at `templates/rendered/_theme/<name>.html` and is regenerated by the
publish pipeline (v1 wave 2 item 7) when a `/_theme/<slot>/` page
publishes.

## 7. Seeded slot content

Two seed Markdown files capture the existing nav.html / footer.html
content so the wave is non-regressive on day 1:

```
content/_theme/header.md
content/_theme/footer.md
```

Authored in this wave with frontmatter that opts-out of the v2 block
parser (no directives), so v1 byte-preservation applies.

The header.md uses link-list syntax that the slot template renders as
nav items; the footer.md is straight prose with link-list and
copyright. The Jinja partials (`nav.html`, `footer.html`) wrap the
rendered HTML in their existing semantic shells.

## 8. Publish pipeline delta

`blueprints/admin/cms/publish.py::publish` for `DIRECT_MD` pages
currently writes the source MD and commits. Theme slots additionally
trigger `regenerate_slot_cache(name)`:

```diff
 def publish(page_id: str) -> PublishReport:
     ref = resolve(page_id)
     ...
     # after git_commit:
+    if page_id.startswith("/_theme/"):
+        from blueprints.admin.cms.theme_slots import regenerate_slot_cache
+        slot = page_id.strip("/").split("/", 1)[1]   # "header" from "/_theme/header/"
+        regenerate_slot_cache(slot)
     return ...
```

The cache is also regenerated unconditionally at app startup so a
fresh checkout has populated cache without needing a publish.

## 9. Verification gate

1. `python -c "from blueprints.admin.cms.resolver import resolve; \
    r = resolve('/_theme/header/'); assert r is not None and r.kind.value == 'direct_md'"`
2. `/admin/edit/?id=/_theme/header/` opens the editor with header.md content.
3. Editing + publishing header.md updates `templates/rendered/_theme/header.html`
   and the public site's nav reflects the change on next request.
4. `git log -- content/_theme/header.md` shows a per-edit commit
   `cms: publish /_theme/header/`.
5. Pre-existing nav links (Home, Projects, Blog, Research, David Simmons,
   Contact, Search) all still render after migration — no regression
   on the public surface.

## 10. Sidebar deferment (v2-6b)

`content/_theme/sidebar.md` is a future addition. Deferred because:
- v1 has no sidebar partial — the slot would render nowhere unless a
  template change is co-deployed.
- Sidebar render placement is template-specific (research index vs blog
  index vs leaf pages) — needs a §11 design pass on which templates
  consume the slot.

PI may schedule v2-6b after v2-3 (block library) lands, since sidebar
content is the natural target for callout / featured-post / contact
blocks.
